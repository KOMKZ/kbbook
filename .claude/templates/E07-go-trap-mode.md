# E07 Go 陷阱实战模板

## 适用场景

- Go 语言中"看起来合理的代码，实际有隐藏 bug"的主题
- 切片、map、interface、channel、defer、error、context 等基础语法的陷阱
- sync 原语的误用场景
- 标准库 API 的常见错误用法

**不适合**：
- "从零实现一个组件"型（需另外的构建型模板）
- runtime 原理深挖型（用 E02）
- 多方案对比选型（用 T03）

## 文章骨架

```text
背景 — 一个具体的开发场景，你在写什么功能

常规做法（有问题）
  → 自然想到的代码
  → 运行，出现什么意外结果
  → 为什么：根因分析（配内存布局图或状态图）

正确做法
  → 方案 A + 方案 B（至少两种修复方式）
  → 验证：原问题不再出现

总结
  → 知识点表格（# + 知识点 + 说明）
  → 面试速记一行

完整可跑代码 + 运行输出
```

## 门禁自检

| # | 检查项 | 通过标准 |
|---|--------|---------|
| G1 | **单场景聚焦** | 整篇只讲一个开发场景，不引入第二个无关案例 |
| G2 | **错误代码可跑** | 错误版本的代码 `go run` 能复现问题，输出和文中一致 |
| G3 | **根因有图** | 内存布局/状态流转/时序至少有一张示意图（ASCII 或 Mermaid） |
| G4 | **至少两种修复** | 正确做法列至少两种，且有代码对照 |
| G5 | **知识点表格化** | 总结用表格，每条一行，有"一句话说明"列 |
| G6 | **代码在文中可独立运行** | 文末完整代码块复制到 `main.go` 后 `go run` 无误 |
| G7 | **背景 ≤ 3 句** | 背景部分不展开，3 句以内交代清楚需求 |
| G8 | **无"为什么重要/为什么必考"段落** | 不写教学引导语句，直接进入场景 |

## 写作约束

### 背景

```markdown
## 背景

你要写一个 [具体功能]。输入 [数据结构]，[做什么操作]，返回 [结果]。
```

上限 3 句。不展开讲"为什么选这个例子"。

### 常规做法（有问题）

```markdown
## 常规做法（有问题）

思路很直接：[一句话说思路]。

[代码块]

来测一下：

[代码块 + 输出，标注异常处]

**为什么会这样**：

[根因分析，配图]
```

关键：
- "为什么会这样"必须配一张内存/状态示意图，不能只有文字
- 图中的关键字段（ptr/len/cap/状态）要标注具体值

### 正确做法

```markdown
## 正确做法

**方案 A**：[一句话思路]

[代码块 + 关键注释]

**方案 B**：[一句话思路]

[代码块 + 关键注释]

**验证**：

[代码块 + 输出，标注修复确认]
```

至少两个方案。如果实在只有一个，写"为什么其他方案不适用"。

### 总结

```markdown
## 总结

| # | 知识点 | 说明 |
|---|--------|------|
| 1 | [概念名] | [一句话] |
| 2 | ... | ... |

**面试速记**：[一句话可背诵的核心结论]
```

知识点列表不解释推导过程（已在正文完成），只保留可回顾的浓缩结论。

### 完整可跑代码

文末放完整 `main.go`，可以直接 `go run`。代码里用注释分段标注"错误做法""正确做法 A""正确做法 B"。后面紧跟 `$ go run main.go` 的实际输出。

## 与 E01 的区别

| 维度 | E01 工程深文 | E07 Go 陷阱实战 |
|------|------------|----------------|
| 场景数 | 多个场景 | 单场景 |
| 篇幅 | 中长（800-1500行） | 短中（200-400行） |
| 结构 | 事故→心智模型→机制→陷阱 | 背景→错误→根因→正确→总结 |
| 验证 | benchmark/pprof/trace | go run 输出 |
| 前置要求 | 需要线上事故/血泪教训 | 不需要 |
