# 调研对比模式

## 适用场景

- 同一问题有多个方案（如 LangChain vs LlamaIndex vs 手写 / vLLM vs llama.cpp / ReAct vs PlanAct）
- 想理解某个领域的演进脉络
- 决策前的方案选型

## 核心约束

| 约束 | 说明 |
|------|------|
| **痛点驱动对比** | 不堆砌特性表格。每个方案先讲它解决了什么痛点，否则对比无意义 |
| **演进而非并列** | 优先按时间/复杂度演进顺序展开，让读者看到"为什么会出现下一个方案" |
| **取舍清晰** | 每个方案的"代价/适用边界"必须明示 |
| **有自己的判断** | 文章必须给出"如果是我，会怎么选" |

## 标准提纲

```markdown
# [领域] 方案对比

> 副标题：从 [最朴素的方案] 到 [当前主流]，每一步在解决什么

> 导语：[一段话说明：这个对比文章不是特性清单，是演进脉络]

## 一、共同要解决的问题
- 这个领域的核心问题是什么
- 评判一个方案好坏的维度有哪些

## 二、朴素方案（baseline）
- 最早/最简单的做法
- 能不能跑？能。够不够好？不够
- 痛点 → 引出下一节

## 三、方案 A
- 出现背景（什么时间、解决什么痛点）
- 核心思想（一句话）
- 关键抽象（必要时给骨架代码）
- 解决了什么 + 还没解决什么
- 适用边界

## 四、方案 B
- 同上结构
- **重点**：相对方案 A 的改进点 + 代价

## 五、方案 C / ...
- 同上

## 六、横向对照
- 用一张表汇总：每个方案在每个维度的表现
- 表格之前必须有"为什么选这些维度"的说明

| 维度 | 方案 A | 方案 B | 方案 C |
|------|-------|-------|-------|
| ...  | ...   | ...   | ...   |

## 七、如果是我会怎么选
- 不同场景下的推荐路径
- 决策树（Mermaid）

## 八、收束
- 这个对比真正教会的判断方式

## 参考
```

## 反例

- ❌ 开头直接放特性对比表 → 读者不知道维度从哪来
- ❌ 罗列每个框架的 API → 没有教学价值
- ❌ 「各有优劣，按需选择」式收尾 → 必须给立场
- ❌ 把所有方案描述成"也行" → 必须指出取舍

## 调研素材要求

写之前 `themes/NN-slug/research/` 必须有：
- 每个方案的官方文档关键章节摘录 + 链接
- 至少看过一份核心源码片段（不要只看 README）
- 真实使用反馈（Issue、博客、HN/Reddit 讨论）
