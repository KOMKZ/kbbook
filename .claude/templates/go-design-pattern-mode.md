# E03 Go 设计模式模板

## 核心立场

Go 设计模式不是把 Java/OOP 模式翻译成 Go。Go 优先:

- 简单具体类型。
- 小接口。
- 组合。
- 显式依赖。
- 函数和结构体直接表达业务。

## 文章骨架

```text
0. 这个模式试图解决什么工程痛点
1. 不用模式时,代码怎么坏掉
2. OOP 版本为什么不适合直接搬进 Go
3. Go 风格的最小实现
4. 什么时候才需要抽象
5. 反模式
6. 项目中怎么落地
```

## 必填对比

每篇必须有三版:

1. 直接写的具体版本。
2. 过度设计版本。
3. Go 风格的平衡版本。

## 自检

- 接口是否由使用方定义?
- 抽象是否来自真实重复?
- 是否能删除一层而不损失表达力?
- 是否把测试和替换需求讲清楚?
