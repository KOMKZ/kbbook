# 推导式写作模式

## 写作原则

| 原则 | 说明 |
|------|------|
| **从问题出发** | 先讲"原来怎么做"，再讲"为什么不够"，最后才讲"新设计是什么" |
| **先动机后抽象** | 每个接口/组件/扩展点先解释解决什么问题，再讲它是什么 |
| **推导而非宣布** | 概念像被需求逼出来的，不要凭空出现 |
| **骨架优先** | 最小可跑骨架先行，工程化细节后置 |
| **节点映射** | 手写骨架最终要对照真实框架/规范 |
| **教学闭环** | 文末必须回到现实：「现在你应该如何重新理解 X」 |

## 文风要求

- **叙事感**：像连载专栏一样推进，有过渡感
- **陪伴感**：常用"现在我们继续往前走一步"这类引导句
- **解释感**：术语首次出现立刻解释
- **结论后置**：重要概念在推导后再下定义
- **避免教条**：少用定义堆砌，多用"如果你来设计会怎么想"

## 必须回答的问题

一篇合格的推导型文章至少回答：

1. 原始做法是什么？
2. 原始做法的痛点是什么？
3. 新抽象解决了哪个具体问题？
4. 为什么不是别的设计，而是现在这个设计？
5. 自己手写的骨架和真实框架怎么对应？
6. 读完后应该如何重新理解这个技术？

## 标准提纲

```markdown
# [主题标题]

> 副标题：[一句话点明：不是讲定义，而是从需求推导]

> 导语：[2-4 句，核心问题 + 推导路径 + 最终收获]

## 开头（Hook）
[描述读者熟悉的困惑、割裂感、学习误区]

## 引言
- 这篇写给谁
- 不打算直接讲什么
- 准备怎样推导
- 读完会得到什么

## 一、原先的做法
- 原来怎么做
- 当时为什么这样做合理
- 给出最小伪代码/极简代码

## 二、痛点
- 规模/复杂度变大后遇到什么问题
- 哪些痛点逼着必须升级

## 三、如果让我设计
- 拆出哪些角色
- 每个角色负责什么
- 给出第一版结构图（Mermaid）

## 四、最小骨架
- 写最原始能跑通的版本
- 明确：这版解决了什么，还没解决什么

## 五、第一次迭代
- 引入第一个关键抽象
- 给出迭代后骨架
- 解释结构升级点

## 六、第二/第三次迭代
- 继续补生命周期/路由/扩展点
- 每个抽象解释设计动机

## 七、回到真实世界
- 手工骨架节点 vs 真实框架节点 对照表
- 真实运行链路图
- 解释真实术语的分工

## 八、重新理解
- 你现在应该如何重新理解 [概念 A]
- 你现在应该如何重新理解 [概念 B]
- 把整条链路连成一句话

## 九、收束
- 如果只记一句话，应该记什么
- 这篇真正教会的学习方式

## 参考
- 规范、官方文档、源码链接、论文
```

优先用 Mermaid：

- **时序图**：before/after、生命周期顺序
- **对照图**：手工骨架 vs 真实框架
- **分层图**：层次职责划分

## 代码示例规则

| 规则 | 说明 |
|------|------|
| **目标** | 展示设计骨架和职责边界，不追求生产可用 |
| **迭代** | 至少"原始版" + "升级版"两份 |
| **拥有权** | 涉及容器/运行时时，代码要体现"谁拥有、谁组织执行、谁管理生命周期" |
| **调用流** | 代码后必须解释执行顺序，给出控制台输出或调用路径 |
| **真跑过** | expected output 必须是真跑出来的，不是想象的 |

## 定稿检查清单

- [ ] 是否从需求/痛点出发，没直接抛定义
- [ ] 每个关键抽象都解释了设计动机
- [ ] 至少一份能跑的最小骨架
- [ ] 解释了代码怎么跑起来
- [ ] 有"手工骨架 vs 真实框架"映射
- [ ] 结尾重建了对核心概念的理解
- [ ] 保留了教程专栏式的叙事感
